Структура репозитория git соответсвует модели gitflow.
1. Формулировка коммита должна представлять из себя описание того, что делает коммит. Пример: "Rafactors methos doSmt() in controller" Глагол должен ьбыть в третьем лице
2. В ветке main находится стабильный код, готовый для выпуска. Код в ней - реззултат объединения release и develop. 
3. В develop - то, что сейсчас является разрабатываемым. Оно отржает текущее состояние разработки проекта и служит основй для других
4. hotfix - ветки, которые  отпочковываются от main
5. release - ветка, в которую вностя окнчательные изменения перед тем, как она будет обхединена с main.
6. ветка feature - содаются при добавлении новых функциональных возможнстей в проект. Начинаются от ветки develop и заканчиваются там же сразу же, как занчивается разработка функциональности в этой вектке.

 Общие соображения
 Процесс работы над фичой
 1. Добавление себе локально всех изменений: git pull develop
 1. Переключение в develop: создание ветки в develop: git checkout develop
 2. Создание ветки с фичой: git checkout [feature-name] (назваине фич, если состои из нескольки слов, иджет через тире)
 3. После того, как была закночнчена работа на фичой, нужно переключиться в develop, слить её с веткой develop, и после этого удалить векту с фичой. git checkout develop; git merge [new feature]; git branch -d [new feature]
 
 Чтобы залить изменения. используется git push.
 
 Все вышеукзанные оперции  можно осуществлять с помощью графического интерфейса в IDE.
 
Подготовка к релизу и релиз
Как только мы хотим выпустить новый релиз нашего приложения, необходимо к нему подготовиться.
Мы должны находиться в ветке develop
Создаем и переключаемся на ветку для подготовки данного релиза (git checkout b release 1.0.0)
Вносим финальные правки и изменения.
Переключаемся на master ветку и объединяем ветку релиза с master веткой (git checkout m aster; git merge release 1.0.01.0.0)
Также помечаем тегом номер релиза для простоты поиска и понимания
Переключаемся в ветку develop и добавляем изменения релизной ветки в ветку для разработки (git checkout develop; git merge release-release version)
Удаляем временную ветку релиза (git branch d release-release version)

Исправление багов после релиза (hotfix):
Если после релиза мы заметили какие-то серьезные баги, которые нужно срочно исправить, то мы прибегаем к хотфиксам. Сам хотфикс
16
представляет из себя срочное исправление критической ошибки или уязвимости в программе.
Создаем и переключаемся на ветку для фикса (git checkout git checkout --b hotfixb hotfix--1.0.11.0.1)
Вносим в ней необходимые исправления для устранения бага
Переключаемся на main ветку и объединяем ветку хотфикса с main веткой (git checkout master; git merge hotfixgit checkout master; git merge hotfix--1.0.11.0.1)
