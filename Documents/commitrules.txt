Структура репозитория git соответсвует модели gitflow модификациями.
1. Формулировка коммита должна представлять из себя описание того, что делает коммит. Пример: "Rafactors methos doSmt() in controller" Глагол должен быть в третьем лице. При работе над фичой коммит должен начинаться с названия ветки в формате feat/feature-name
2. В ветке main находится стабильный код, готовый для выпуска. Код в ней - результат объединения release и develop
3. В develop - то, что сейсчас является разрабатываемым. Оно отражает текущее состояние разработки проекта и служит основой для других
4. hotfix - ветки, которые  отпочковываются от release
5. release - ветка, в которую вносятся окнчательные изменения.
6. ветка feature - создаются при добавлении новых функциональных возможностей в проект. Начинаются от ветки develop и заканчиваются там, как заканчивается разработка функциональности в этой вектке.

 Общие соображения
 Процесс работы над фичой
 1. Добавление себе локально всех изменений: git pull develop
 1. Переключение в develop: git checkout develop 
 2. Создание ветки с фичой: git checkout [feature-name] (назваине фич, если состоит из нескольких слов, идет через тире)
 3. После того, как была закончена работа на фичой, нужно переключиться в develop, слить её с веткой develop, и после этого удалить веткой с фичой. Команды: git checkout develop; git merge [new feature]; git branch -d [new feature]
 4. Для вливания фичи нужно на GitHub создать pull request, который должен одобрить один из разарботчиков  
 Чтобы залить изменения используем git push.
 
 Все вышеукзанные операции  можно осуществлять с помощью графического интерфейса в IDE.
 
Подготовка к релизу и релиз
Как только мы хотим выпустить новый релиз нашего приложения, необходимо к нему подготовиться.
Мы должны находиться в ветке develop
Создаем и переключаемся на ветку для подготовки данного релиза (git checkout b release 1.0.0)
Вносим финальные правки и изменения.
Переключаемся на main ветку и объединяем ветку релиза с main веткой (git checkout main; git merge release 1.0.01.0.0)
Также помечаем тегом номер релиза для простоты поиска и понимания
Переключаемся в ветку develop и добавляем изменения релизной ветки в ветку для разработки (git checkout develop; git merge release-release version)
Удаляем временную ветку релиза (git branch d release-release version)

Исправление багов после релиза (hotfix):
Если после релиза мы заметили какие-то серьезные баги, которые нужно срочно исправить, то мы прибегаем к хотфиксам. Сам хотфикс
представляет из себя срочное исправление критической ошибки или уязвимости в программе.
Механизм:
Создаем и переключаемся на ветку для фикса (git checkout git checkout --b hotfixb hotfix--1.0.11.0.1)
Вносим в ней необходимые исправления для устранения бага
Переключаемся на main ветку и объединяем ветку хотфикса с main веткой (git checkout master; git merge hotfix git checkout master; git merge hotfix--1.0.11.0.1)
